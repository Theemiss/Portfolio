Overview

DarDev Studio is a dev orchestration brain for engineering teams. Not a chat wrapper on a demo API. I designed and built it as architecture owner at DarDev, validated against our own monorepos: ingest docs and code, route intents, delegate specialist subtasks, and ship through patch and CI gates.

The platform ships under Apache-2.0. A scrubbed OSS export is in active release on GitHub while the full monorepo stays private during hygiene review.

Vision

Most teams bolt an LLM onto an IDE and call it "AI-assisted development." DarDev Studio treats AI orchestration as platform infrastructure: explicit memory, retrieval quality, tool boundaries, eval gates, and inference routing that teams can own and extend.

The north star is an engineering brain that knows your org context (docs, code graph, session history), routes work to the right specialist mode without manual recipe picking, and only ships when offline evals pass. Open source so teams can inspect, fork, and run it on their stack - local models, cloud APIs, or edge workers.

DarDev Studio is not a consumer chat product and not an education LMS. It is infrastructure for teams that already ship production software and want agentic workflows with the same rigor they apply to CI/CD.

Agents that ship code need eval gates, not vibes. DarDev Studio is built for teams that would not merge without tests.

The challenge

Engineering teams adopting LLMs hit the same wall: retrieval quality is inconsistent, context windows get stuffed, agents loop without quality gates, and "AI features" ship as thin API wrappers instead of integrated workflows.

DarDev Studio treats orchestration as infrastructure: explicit memory tiers, hybrid retrieval, bounded tool execution, and offline eval suites before changes land in the dev-team profile.

Five memory tiers

Context is split by mechanism, not one giant prompt. Session history and episodic summaries cover the current thread. Org RAG runs BM25, lexical, and semantic retrieval over docs and code corpora with BGE-M3 embeddings stored in SQLite with sharded chunk indexes. A code graph tier resolves symbols and imports. Web search adds ephemeral external context. A planner tier delegates subtasks to specialist modes.

Each tier has its own retrieval path and prompt budget. The orchestrator selects which tiers to activate per intent instead of dumping everything into one context window.

IT orchestrator walkthrough

In dev-team mode (PRODUCT_PROFILE=dev-team), a natural-language goal enters the IT orchestrator. The intent router selects memory tiers and tools. prepare-turn runs hybrid RAG retrieval, applies prompt budget rules, and registers patch context. The agent loop executes up to five iterations with a maximum of three tools per round, streaming responses with retries. Tools include code graph lookup, workspace patches, web search, and delegate_subtask for parallel specialist work.

On failure the orchestrator can reprompt and re-route rather than silently hallucinating forward. Offline eval gates cover orchestrator intents, RAG IT quality, and graph lookup before integration merges.

Quality is enforced by eval suites, not by hoping the model behaves. npm run eval:orchestrator-intents and eval:rag:it are part of the ship path.

Inference plane

Inference is pluggable, not locked to one vendor. Model packs M1 through M5 route requests across local GGUF via llama.cpp, LiteRT/Gemma, Qwen tiers, Ollama, OpenAI-compatible HTTP endpoints, and edge workers on Android and Windows. Provider failover and an embed pool keep retrieval and generation resilient when a backend is down or cost-sensitive routing is needed.

Eval and quality gates

Agentic systems without evals are demos. DarDev Studio ships with offline gates that must pass before orchestrator and RAG changes merge into the dev-team profile.

  • eval:orchestrator-intents — intent routing accuracy across representative engineering goals
  • eval:rag:it — retrieval quality on IT/domain corpora with citation verify
  • eval:graph — code graph lookup correctness
  • eval:plan-mode and adaptive orchestrator matrices
  • validate:orchestrator-prompts — API prompt matrix consistency
  • npm test — core engine unit coverage on mock runtime without requiring local GPU weights
Interview walkthrough: architecture doc, one prepare-turn flow, orchestrator mode, then eval output. Fifteen minutes of systems depth, not a slide deck.

Key decisions

Memory tiers over one context blob

Each tier has a separate retrieval and budget mechanism. Mixing session, corpus, graph, and web into a single prompt creates silent quality loss. Tiers make tradeoffs explicit.

Offline eval gates before merge

Orchestrator intents, RAG IT suites, and graph lookup evals run in CI-style gates. Agentic systems without evals are demos, not platforms.

Bounded agent loops

Max five iterations and three tools per round prevent runaway token spend and infinite tool chatter. Streaming and retries stay inside those bounds.

Open source as distribution

Apache-2.0 on GitHub is the front door. No waitlist theater. Teams inspect the orchestrator, eval runners, and architecture docs directly.

Production validation on DarDev engineering workflows

The platform ingests our own docs and code corpora and routes real patch workflows. Not a sandbox disconnected from delivery.
