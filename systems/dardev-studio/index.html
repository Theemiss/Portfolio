Overview

DarDev Studio is a dev orchestration brain for engineering teams — not a chat wrapper on a demo API. I designed and built it as architecture owner at DarDev to dogfood our own monorepo: ingest docs and code, route intents, delegate specialist subtasks, and ship through patch and CI gates.

It is a separate product from DarDevLab, which is project-based engineering education on dardevlab.com. DarDevLab teaches learners; DarDev Studio orchestrates how engineers work with AI on real codebases.

The platform ships under Apache-2.0. The public GitHub export is in active release — architecture and eval patterns are documented here while the OSS slice completes review.

The challenge

Engineering teams adopting LLMs hit the same wall: retrieval quality is inconsistent, context windows get stuffed, agents loop without quality gates, and "AI features" ship as thin API wrappers instead of integrated workflows.

DarDev Studio treats orchestration as infrastructure — explicit memory tiers, hybrid retrieval, tool execution with limits, and offline eval suites before changes land in the dev-team profile.

Five memory tiers

Context is split by mechanism, not one giant prompt. Session history and episodic summaries cover the current thread. Org RAG runs BM25, lexical, and semantic retrieval over docs and code corpora with BGE-M3 embeddings. A code graph tier resolves symbols and imports. Web search adds ephemeral external context. A planner tier delegates subtasks to specialist modes.

IT orchestrator walkthrough

In dev-team mode, a natural-language goal enters the IT orchestrator. The intent router selects memory tiers and tools. prepare-turn runs hybrid RAG retrieval, applies prompt budget rules, and registers patch context. The agent loop executes up to five iterations with a cap on tools per round, streaming responses with retries. Tools include code graph lookup, workspace patches, web search, and delegate_subtask for parallel specialist work. Offline eval gates cover orchestrator intents, RAG IT quality, and graph lookup before integration merges.

Quality is enforced by eval suites, not by hoping the model behaves — npm run eval:orchestrator-intents and eval:rag:it are part of the ship path.

Core systems

  • Hybrid RAG — BM25 + lexical + semantic retrieve; sharded chunk indexes and citation verify
  • Agent loop — multi-iteration LLM execution with tool calls, streaming, and bounded retries
  • IT orchestrator — intent routing, tier selection, parallel delegates, reprompt on failure
  • Turn preparation — RAG retrieve, episodic memory, prompt budget, patch registry
  • Tool registry — graph_lookup, workspace_patch, web_search, delegate_subtask, browser tools
  • Inference plane — LiteRT/Gemma, llama.cpp, Qwen tiers, Ollama, OpenAI-compatible HTTP, edge workers
  • Eval suite — orchestrator intents, RAG IT, graph, plan mode, adaptive orchestrator, API prompt matrix
  • AI-first dev workflow — dogfood mirror, plan export, patch pipeline, platform TUI on own monorepo

Key decisions

Memory tiers over one context blob

Each tier has a separate retrieval and budget mechanism. Mixing session, corpus, graph, and web into a single prompt creates silent quality loss — tiers make tradeoffs explicit.

Offline eval gates before merge

Orchestrator intents, RAG IT suites, and graph lookup evals run in CI-style gates. Agentic systems without evals are demos, not platforms.

Dogfood on DarDev engineering work

The platform ingests our own docs and code corpora and routes real patch workflows — not a sandbox disconnected from delivery.

Split from DarDevLab in product and repos

Education and orchestration serve different buyers and evidence tiers. Conflating them weakens both stories and confuses recruiters reviewing DarDev work.
